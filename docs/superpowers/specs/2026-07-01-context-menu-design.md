# Right-Click Context Menu — Design

## Purpose

Expose the extension's three existing keyboard-shortcut actions through a
native Chrome right-click context menu, as an alternative trigger for users
who don't want to configure or remember a keyboard shortcut.

## Scope

- Menu appears only on the 5 supported job sites (linkedin.com, indeed.com,
  glassdoor.com, wellfound.com, welcometothejungle.com) — matches the
  existing `content_scripts.matches` patterns in `manifest.json`.
- Menu shows all 3 actions unconditionally, regardless of whether the
  corresponding keyboard shortcut is configured/cleared in the options page.
- Actions are grouped under a single parent "JD Grab" submenu rather than
  as 3 separate top-level context menu entries.
- No changes to keyboard shortcut behavior — this is a purely additive
  trigger path.

## Manifest changes

Add `"contextMenus"` to `permissions` in `manifest.json`.

## Menu registration

In `background.js`, on `chrome.runtime.onInstalled`, register:

- Parent item: `id: "jd-grab"`, title `"JD Grab"`, `contexts: ["page"]`
- Three children, each with `documentUrlPatterns` set to the same 5 host
  patterns used in `content_scripts.matches`:
  - `id: "select-description"` — title "Select Job Description"
  - `id: "open-title"` — title "Open Job Title in New Window/Tab"
  - `id: "open-title-and-select"` — title "Open Job Title & Select Description"

## Click handling

In `background.js`, `chrome.contextMenus.onClicked` listener maps each
child `id` to a message sent to the active tab's content script:

```js
chrome.tabs.sendMessage(tab.id, { action: 'runContextMenuAction', name: <name> });
```

Where `<name>` is one of `selectDescription`, `openTitle`,
`openTitleAndSelect`.

## Content script changes

`content.js` already has three functions, decoupled from the keydown event
object, that implement each action:

- `selectAboutTheJobSection()`
- `openJobTitleLink()`
- `openAndSelectInNewWindow()`

Add a new branch to the existing `chrome.runtime.onMessage` listener
(alongside the existing `selectJobDescription` handler) that dispatches by
`name`:

```js
if (message.action === 'runContextMenuAction') {
  if (message.name === 'selectDescription') selectAboutTheJobSection();
  if (message.name === 'openTitle') openJobTitleLink();
  if (message.name === 'openTitleAndSelect') openAndSelectInNewWindow();
}
```

No refactor of the existing functions is needed — they already read from
`currentOpenTarget`/DOM independent of how they're invoked.

## Error handling

If the content script isn't injected/ready when a menu item is clicked
(rare, since the menu itself is scoped to matching URLs), `sendMessage`
fails silently — same failure mode as the existing keyboard-shortcut path.
No new error surface is introduced.

## Testing

- No automated coverage: native Chrome context menus are OS-level UI, not
  part of the page DOM, so Playwright cannot drive or screenshot them.
- Manual verification required: right-click on each of the 5 supported
  sites, confirm the "JD Grab" submenu appears with all 3 items, and that
  each behaves identically to its keyboard-shortcut counterpart.

## Store assets

The context-menu screenshot (showing the native right-click menu open on a
job page) cannot be generated by `scripts/generate-screenshots.mjs` for the
same reason automated testing is out — it's OS chrome, not page content.
This screenshot must be captured manually (open a real job page, right-click,
screenshot the OS-native menu) before it can be added to
`store-assets/screenshots/`.

## Versioning

Bump the minor version (1.5.1 → 1.6.0) in `manifest.json` and
`package.json` as part of this change, per semver (new user-facing
feature, no breaking changes).
